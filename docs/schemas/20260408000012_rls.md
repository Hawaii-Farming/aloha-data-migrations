# Row-Level Security

All Row-Level Security policies and the one helper function they depend on live in a single file:

```
supabase/migrations/20260401000200_sys_rls_policies.sql
```

The file sits at the end of the migration chain on purpose — every table referenced by a policy must exist first, and the helper queries `hr_employee`, so it has to come after that table is created.

## Convention: SELECT-only RLS, writes via service_role

Every org-scoped table gets the **same** three-statement block:

```sql
ALTER TABLE public.<table> ENABLE ROW LEVEL SECURITY;

CREATE POLICY "<table>_read" ON public.<table>
  FOR SELECT TO authenticated
  USING (org_id IN (SELECT public.get_user_org_ids()));

GRANT SELECT ON public.<table> TO authenticated;
```

There are **no** `INSERT`, `UPDATE`, or `DELETE` policies on any org-scoped table. Every mutation flows through the `service_role` key in a server-side route action, and granular CRUD (`can_create` / `can_edit` / `can_delete` / `can_verify`) is enforced in the app layer via `hr_module_access` before the write is issued. Direct PostgREST writes from the browser session client are blocked because no policy permits them.

The two historical exceptions that once let the session client write time-off requests and performance reviews directly were removed — every table now follows the same pattern with no exceptions.

## One root exception: `org`

`org` is the root tenant table itself; it doesn't have an `org_id` column (it has `id`), so the read policy is keyed on `id` instead:

```sql
CREATE POLICY "org_read" ON public.org
  FOR SELECT TO authenticated
  USING (id IN (SELECT public.get_user_org_ids()));
```

Everything else uses the standard `org_id IN (...)` predicate.

## Helper: `get_user_org_ids()`

```sql
CREATE OR REPLACE FUNCTION public.get_user_org_ids()
RETURNS SETOF TEXT
LANGUAGE sql
STABLE
SECURITY DEFINER
SET search_path = public
AS $$
  SELECT org_id FROM public.hr_employee
  WHERE user_id = auth.uid()
    AND is_deleted = false;
$$;

GRANT EXECUTE ON FUNCTION public.get_user_org_ids() TO authenticated;
```

Returns the set of `org_id` values the currently authenticated user belongs to, resolved via `hr_employee.user_id = auth.uid()`. Used by every `read` policy in the file.

### Why `SECURITY DEFINER`

`hr_employee` itself has RLS enabled. If the helper ran with the caller's permissions, the policy on `hr_employee` would call `get_user_org_ids()` → which queries `hr_employee` → which re-triggers its own policy → infinite recursion. `SECURITY DEFINER` runs the function as the schema owner with RLS bypassed, breaking the loop.

### `STABLE` — not `IMMUTABLE`

The function reads from `hr_employee`, whose contents can change between statements. `STABLE` tells the planner the function returns the same result within a single query but may differ across queries.

## Coverage

100 RLS-enabled tables at the time of writing:

- 1 root table: `org`
- 99 org-scoped tables across every module (org_*, hr_*, sys_navigation derivatives, grow_*, invnt_*, ops_*, sales_*, pack_*, maint_*, fsafe_*, fin_*).

The `sys_*` tables (`sys_uom`, `sys_access_level`, `sys_module`, `sys_sub_module`) are **not** RLS-enabled because they are global reference data shared across all orgs. Same for `grow_pest` and `grow_disease`.

## Adding a new table

When you add an org-scoped table, append a block to `20260401000200_sys_rls_policies.sql` in the position that matches the table's slot in the migration chain (so policies file stays in a predictable order). Then re-run `supabase db push` — Supabase CLI only applies incremental changes, and the policies file re-runs every time thanks to `CREATE OR REPLACE FUNCTION`.

## Adding a new module? Re-issue the GRANT

PostgREST's `authenticated` role starts with no privileges on new tables. The `GRANT SELECT` in the block above is what lets the browser session client read rows after RLS has filtered them — without it, even rows that pass RLS return as empty.
