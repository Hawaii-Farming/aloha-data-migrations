---------------------------------------------------------

Intuit requires both URLs for production approval. They must return
HTTP 200 with relevant content (Intuit's automated checker hits them).

aloha-app has no legal pages today. Recommended setup — two new public
routes in aloha-app:

  app/routes/legal/eula.tsx
  app/routes/legal/privacy.tsx

Add to app/routes.ts (alongside the existing rootRoutes):

  route('legal/eula',    'routes/legal/eula.tsx'),
  route('legal/privacy', 'routes/legal/privacy.tsx'),

Each can render a single markdown file (or just plain JSX) — no auth,
no DB access. Then on the Intuit form:

  End-user license agreement URL: https://aloha.ag/legal/eula
  Privacy policy URL:             https://aloha.ag/legal/privacy

Content scope: identify the operator (Hawaii Farming), describe what
data the app collects/processes, list third-party processors (Supabase,
Google Cloud, Anthropic, Intuit), state retention/deletion policy, and
provide a contact email for data requests. Have your legal advisor
review before submitting for production approval.

Lighter alternative: drop static HTML at public/legal/eula.html and
public/legal/privacy.html so Cloud Run serves them without going
through React Router. Faster to ship; styling won't match app shell.

3. WHAT EACH ROUTE DOES
------------------------

connect.ts
  Where users click "Connect to QuickBooks". Build the Intuit authorize
  URL and 302-redirect:

    GET https://appcenter.intuit.com/connect/oauth2
        ?client_id=<INTUIT_CLIENT_ID>
        &response_type=code
        &scope=com.intuit.quickbooks.accounting
        &redirect_uri=https://aloha.ag/api/quickbooks/callback
        &state=<random-anti-csrf-string-stored-in-session>

  Use scope = com.intuit.quickbooks.accounting (add openid+profile+email
  if you also want OpenID claims). Generate a random `state` and stash
  it in a session/cookie so callback.ts can verify it.

callback.ts
  Intuit redirects here after the merchant authorizes. Query string:

    ?code=<auth_code>
    &state=<echo>
    &realmId=<QB-company-id>

  Verify state matches the value you stored. Then POST to:

    POST https://oauth.platform.intuit.com/oauth2/v1/tokens/bearer
    Authorization: Basic base64(CLIENT_ID:CLIENT_SECRET)
    Content-Type:  application/x-www-form-urlencoded
    Body:
      grant_type=authorization_code
      &code=<auth_code>
      &redirect_uri=https://aloha.ag/api/quickbooks/callback

  Response:
    {
      "access_token":  "...",
      "refresh_token": "...",
      "expires_in":    3600,        // seconds (~1 hour)
      "x_refresh_token_expires_in": 8726400  // ~101 days
    }

  Store realmId + access_token + refresh_token + expiry in a new table
  (proposed: org_quickbooks_token, one row per org). Redirect back to
  settings with a success flash message.

disconnect.ts
  Two ways the user lands here:

    a) From within aloha-app — user clicks "Disconnect QB" in settings.
       Revoke the refresh token (POST to
       https://developer.api.intuit.com/v2/oauth2/tokens/revoke with
       Basic auth + body `token=<refresh_token>`), delete the local
       org_quickbooks_token row, redirect to settings.

    b) From QB itself — Intuit fires a server-to-server call here when
       the merchant disconnects in QuickBooks UI. Same cleanup: delete
       the local org_quickbooks_token row. Respond 200 OK.

5. SUPABASE TABLE FOR TOKEN STORAGE
------------------------------------

Proposed schema for a new migration in this repo
(supabase/migrations/<timestamp>_org_quickbooks_token.sql):

  CREATE TABLE IF NOT EXISTS org_quickbooks_token (
      org_id              TEXT NOT NULL REFERENCES org(id),
      realm_id            TEXT NOT NULL,
      access_token        TEXT NOT NULL,
      refresh_token       TEXT NOT NULL,
      access_expires_at   TIMESTAMPTZ NOT NULL,
      refresh_expires_at  TIMESTAMPTZ NOT NULL,
      connected_by        TEXT,                       -- hr_employee.id
      connected_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
      updated_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
      is_deleted          BOOLEAN NOT NULL DEFAULT false,
      PRIMARY KEY (org_id)
  );

  -- Service-role only: tokens never exposed via PostgREST to authenticated.
  -- (Don't add a SELECT policy.)

The composite-PK pattern we just adopted means we can have separate QB
connections per org if we ever multi-tenant beyond Hawaii Farming.


6. UPLOAD FLOW (later)
-----------------------

The sales_po table already has qb_uploaded_at + qb_uploaded_by columns.
Once OAuth is wired up, the upload action would:

  1. Read org_quickbooks_token; refresh access_token if expired.
  2. POST invoice batch to QB Online (Invoice + Customer endpoints).
  3. On success, set sales_po.qb_uploaded_at = now() and
     sales_po.qb_uploaded_by = current employee id.

Token refresh (when access_token is expired but refresh_token is not):

  POST https://oauth.platform.intuit.com/oauth2/v1/tokens/bearer
  Authorization: Basic base64(CLIENT_ID:CLIENT_SECRET)
  Body: grant_type=refresh_token&refresh_token=<refresh_token>

  Response includes a NEW refresh_token (rotation). Store both.
